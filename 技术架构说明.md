# 药典问答知识库（RAG）项目技术架构与方案说明

本文档详细描述了本项目（药典检索大模型问答应用 RAG_project_zhyd）的技术栈、工作流程及核心实现原理。此文档旨在帮助了解本项目背后的设计思路、开发工具集及各个模块之间的协同工作方式。

## 1. 核心技术栈概览

本项目是一个基于**检索增强生成 (Retrieval-Augmented Generation, RAG)** 架构的本地知识库问答系统，主要面向《中华人民共和国药典》的使用场景。

- **编程语言**: Python 3 (>=3.7)
- **大语言模型 (LLM) 接口**: OpenAI API (可兼容任何支持 OpenAI 格式的本地或云端大模型接口如智谱、Qwen 等)
- **文本向量化模型 (Embedding)**: `sentence-transformers` (`paraphrase-multilingual-MiniLM-L12-v2`模型)
- **向量检索引擎**: `FAISS` (Facebook AI Similarity Search - CPU版)
- **大文本存储检索数据库**: `Elasticsearch` (版本 7.17.21)
- **Web 前端交互界面**: `Gradio` (轻量级机器学习模型演示和交互前端库)
- **文档提词处理**: `python-docx`
- **数值计算及矩阵操作**: `numpy`

## 2. 核心模块与实现细节

项目整体可以分为以下几个核心模块：数据导入与切分、文本嵌入与存储、相似度检索与上下文构建、大模型推理与交互。

### 2.1 文档解析与分块 (Document Parsing & Chunking)
- **功能描述**: 用户通过界面的“文档导入”功能上传包含医学药典信息的 Word 文档 (`.docx`)。
- **技术方式**: 
  - 使用 `python-docx` 解析文档的段落和文字样式。
  - **基于样式的切分策略**: 系统会识别文档中特定段落字体大小（如 12pt）、特定标点符号规则（如匹配 `【...】` 等特定括号形式的小标题）。提取出完整的“篇章（对应具体药品）”以及下属的“小节（如处方、制法、性状、功能主治等字段）”。
  - **中文名清洗**: 使用正则表达式 `[\u4e00-\u9fff]+` 从文档的提取内容中清洗并提取标准的中文标题。

### 2.2 存储方案：双库协同 (Elasticsearch + FAISS)
项目采用了“倒排索引引擎”加“纯向量数据库”配合的混合存储思路。

- **全文存储大库 (Elasticsearch)**:
  - 系统内置了一个 `start_es.bat` 脚本用于在本地一键启动已配置好的 Elasticsearch 服务。
  - 当文档切分成篇章后，应用通过 `elasticsearch` 的 Python 客户端，按 `(id=药品名, content=完整的篇章内容)` 的格式创建倒排索引，持久化保证原始数据的完整性与安全性，为后续内容重构和验证提供基准。
- **向量检索库 (FAISS)**:
  - 核心处理模块从 Elasticsearch 提取原始切分数据，并对抽取出的每个小节（如“薄荷”的“功能主治”片段）进行独立向量化。
  - 使用高密度的本地内存级向量相似性搜索库 `FAISS` (`IndexFlatL2` 索引，L2距离度量方案) 对向量化的片段进行管理。
  - 数据保存于本地的压缩阵列文件 `.npz` 中，其中囊括了：向量数组(embeddings)、文档对应ID、包含小标题与明细文本的元信息(texts)，这使得系统不仅可以快速检索，还可以实现内存的快速加载与离线落盘保存。

### 2.3 文本向量化与特征工程 (Embedding Pipeline)
- **模型选择**: 应用了轻量级、支持多语言特性的模型 `paraphrase-multilingual-MiniLM-L12-v2`。
- **技术方式**: 
  - 使用 `SentenceTransformer` 加载本地权重/在线下载。
  - 无论是在写入库（建库）时还是用户提问（查询）时，都调用 `model.encode(..., convert_to_numpy=True)` 将自然语言转化为等维度的高维密集向量 (Dense Vector)，供后续对比使用。这个模型在计算效率与多语言语义捕捉上取得了较好的平衡。

### 2.4 信息抽取与标准化智能体 (Information Standardizer)
在项目中利用大语言模型的能力进行了类似 Agent（智能体）的数据清洗和提取包装。
- **工具实现**: 设计了 `MedicineInfoStandardizer` 药物信息标准化器。
- **技术方式**: 
  - 定义了一个完备的药典关键字段列表（如：处方、制法、性状、鉴别、功能主治等近30种）。
  - 通过精确设计的提示词（Prompt Template），引导大语言模型判断并抽取用户问题或某一段无序的文本片段里到底映射到了哪些标准字段。
  - **意图流控 (Query Classification)**: 在检索开启前，应用会调用 `classify_pharmacy_query`，利用 LLM 进行问题分类，精准判别用户问题是否为“药学相关问题（good）”还是非药学相关的闲聊（bad），来规避系统回答无效问题。

### 2.5 检索生成引擎 (RAG Engine Core)
系统回答用户问题的具体生命周期为：
1. **意图获取与向量化**: 接收到用户的自然语言问题（如：“复方薄荷脑滴鼻液的处方是什么？”），首先调用向量化模型将该问题转换为高维向量。
2. **Top-K 召回**: 向 FAISS 本地索引库发起查询指令 `index.search(query_embedding, top_k=3)`，计算 L2 欧式距离，提取相关度最高的前 3 个原始药典片段（包含所属药品名、该小片段标题、具体内容）。
3. **上下文整合**: 将召回的多个零散片段拼装统一，格式化为带有【来源标识】的长文本 Context。
4. **Prompt 构建与推断**: 运用定制化的 System Prompt（*“你是一个专业的药典问答助手。请根据提供的上下文回答...”*）将用户原问题和检索到的上下文一并提交给由 OpenAI 接口适配的语言大模型。
5. **流式输出 (Stream Streaming)**: 开启 `stream=True` 并使用生成器 `yield` 机制。大模型边推理边逐字将答案展示给前端用户，避免用户产生过长的加载等待。

### 2.6 用户界面交互系统 (Gradio UI/UX)
为了便于非技术人员使用，本系统完全舍弃了命令行，设计了可视化的操作面板。
- **多 Tab 面板结构**: 包含“药典问答”、“文档导入”和“配置”三个选项卡，形成完整的业务闭环。
- **技术方式**: 
  - 基于 `gradio.Blocks` 提供的灵活性，将多项功能整合。
  - **问答面板 (`ChatInterface`)**: 配置了类似主流大模型客户端的对话式 UI，支持历史上下文记录回显；
  - **上传面板**: 支持直传文件和在线配置索引及向量库存储路径，自动触发后端的文档分析流水线。
  - **统一配置页**: 可以直接在界面修改 Elasticsearch 地址、账号、LLM 模型参数、向量库导出位置，并将状态通过全局可变的单例设计方案 (`config.py` 中的 config 类) 实时应用到后端应用代码中，免需重启系统。

## 3. 安装与运行自动化方案
- 项目包含了 `setup.py` 用于快速安装并隔离所有依赖的 Python 库。
- 设置了 `run_rag.bat` 和 `start_es.bat` 脚本作为 Windows 系统的应用启动器，其内部封装了虚拟环境 (`venv`) 路径映射检查和 `KMP_DUPLICATE_LIB_OK=TRUE` 环境变量兜底，以保证运行过程中不发生底层动态库冲突导致程序崩溃。

## 4. 总结
本项目通过“Elasticsearch + FAISS”的双轮驱动实现了本地高效率、强稳健的大体量医学药典知识库检索引擎，结合 `Sentence-Transformers` 多语言模型与强大的大模型逻辑理解能力，并且辅以轻量级的 `Gradio` Web 框架完成了全套且开箱即用的落地实施。在垂直度要求较高、容错率低的医疗/药典类自然语言交互场景下提供了优秀的解决方案架构。
